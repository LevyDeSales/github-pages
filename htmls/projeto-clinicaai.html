Projeto Clinica AI-Swarm V3

WhatsApp ⊂ n8n ⊂ Supabase — atendimento, vendas e follow-up autônomos, auditáveis e escaláveis.

0 ▪ Objetivo-chave

Converter conversas WhatsApp em consultas confirmadas com mínima intervenção humana, mantendo tom humano, velocidade robótica e rastreabilidade médica/LGPD.

Orquestração n8n (2 containers HA) cron, workers paralelos, HTTP WA

3 ▪ Swarm de Agentes (10 especialistas + 1 Writer)

Ordem típica Agente Propósito Exemplo de writer_brief
1 apresentador Boas-vindas, tour inicial mission:"Saudar, perguntar se quer conhecer nossa clínica." tone:"friendly"
2 sdr Qualificação SPIN, quebra objeção mission:"Explorar dor sobre aparelho, mostrar implicações e necessidade."

4 infos_doutores Perfil profissional lista Dr. Ana (CRO …), fotos
5 logística Como chegar, estacionamento texto mapa + Waze link
6 faq_geral Respostas curtas a perguntas comuns "Sim, aceitamos Amil Dental."

8 remarcar Sugerir novos horários idem

Qualquer fluxo usa subconjunto desses agentes em cadeia; o Orquestrador decide.

5 ▪ Fluxos de Alta (exemplos)

5.1 1º Contato → Agendamento

  1. Webhook grava msg.
  2. Orquestrador gera apresentador todo + bloqueia sdr.
  3. apresentador → Writer → WA (saudação).
  4. Lead responde; flag reply OFF.
  5. Orquestrador solta sdr.
  6. sdr cria brief SPIN → Writer envia texto de convencimento.
  7. Lead "quero marcar"; Orquestrador gera agendar todo.
  8. agendar → Writer com 3 slots.
  9. Lead escolhe slot; Orquestrador cria agendar confirmação → Writer.
  10. conversation_state.state = agendado.

5.2 Follow-up Automático

  • worker writer.followup roda via cron n8n a cada 15 min.
  • Consulta conversation_state onde waiting_user_reply = TRUE e next_follow_up_at <= now().
  • Gera brief quick → Writer envia lembrete.
  • Avança follow_up_stage e recalcula next_follow_up_at.

6 ▪ Mecanismos Técnicos

  • Leader election: Orquestrador loop só roda se pg_try_advisory_lock(42).
  • Escalonamento: workers clonáveis; SQL FOR UPDATE SKIP LOCKED evita corrida.
  • Notificações: trigger notify_new_task() → pg_notify → n8n WebSocket, acorda workers instant.
  • Timeout: cada agente executa máx 60 s; se exceder grava status=retry.
  • Retries: até 3; se falhar, cria task human_takeover.

8 ▪ Observabilidade (KPIs)

  • t_first_response – median ≤ 5 s
  • queue_latency – 95p ≤ 2 s
  • %retry_per_agent – < 5 %
  • lead→agenda – ≥ 40 %
  • followup_recovery – + 15 % leads resgatados

Grafana painéis: filas, erros, Postgres locks, WA throughput.
